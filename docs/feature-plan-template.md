> **このファイルは設計ドキュメントのテンプレートです**。新しい設計プランを起こす際は本ファイルをコピーし、`{プレースホルダ}` を埋める / 引用ブロック内のガイドを削除する形で使う。完了したプランは `docs/<topic>.archive.md` にリネームしてアーカイブする（例: `docs/archive/mdxg-rendering-code-block.archive.md`）。

> **書き方の原則**：
>
> - **判断の理由を残す**：何を採用したか だけでなく、なぜ他案を採らなかったかを「設計判断」の比較表に残す。レビュー時 / 後日の見直し時に「同じ議論を繰り返さないため」が目的
> - **DESIGN.md との関係**：プラン完了時に DESIGN.md のどの章を書き換えるかを §9 で明示する。本ドキュメントは寿命のあるプランで、DESIGN.md が永続資料
> - **粒度**：Step は 1 PR / 1 セッションで完了する程度を 1 単位にする。順序依存があれば §4 の構造で明示
> - **コメントは WHY のみ**：プラン本文に「現在のタスク・修正経緯」を書かない（コメントポリシーは `AGENTS.md`）

DESIGN.md {§X.Y} の {対応項目名} に対応するための設計判断と実装手順をまとめる。完了時点で本ドキュメントは DESIGN.md {§X} 表の「{対象機能} = 準拠 / 実装済み」に置換され、本ファイルはアーカイブされる想定（`docs/archive/<topic>.archive.md`）。

## 1. 対応スコープ

> **書く内容**：このプランで何を満たすかを要件単位で表にする。要件は外部規格 / 既存設計ドキュメント / Issue / フィードバックなど **客観的に検証可能なソース** を引用する。「現状」列で start ライン、「完了条件」列で finish ラインを書き、レビュー時のチェックリストとして機能させる。
>
> スコープ外項目（隣接タスクで扱う / 意図的に割り切る）も別ブロックで明示する。「何をやらないか」を書くことで PR レビュー時の議論を絞れる。

{規格 / 既存設計} の N 要件をすべて満たす。

| 要件                     | 現状      | 完了条件                         |
| ------------------------ | --------- | -------------------------------- |
| [MUST] {要件 1 を簡潔に} | 未 / 部分 | {完了状態の判定方法を具体的に}   |

| [SHOULD] {要件 3}        | 未        | {実装アプローチを 1 行で}        |

追加実装（規格上は SHOULD 未満だが UX 上有用）：

- {追加項目 1 の概要と動機}
- {追加項目 2}

## 4. 実装ステップ

> **書く内容**：依存関係順に Step 1, 2, ... を並べる。各 Step は「1 PR で完了 / レビューできる粒度」を目安にし、「成果物」で出力（コードファイル名 / コミットの diff の範囲）を明示する。
>
> Step 内の作業は箇条書き。in-source test の追加 / 手動チェックの追加もこの中で言及する。Step 間の依存（A の完了が B の前提）は冒頭で明示。
>
> **進捗マーカーの付け方**：Step が完了 / 進行中になったら、見出しに `(完了済み)` / `(進行中)` を入れ、本文先頭に `**状態**:` 行を 1 つ追加する。番号は欠番扱いにせず順序付きのまま保ち、後続 Step の番号を詰めない（他節からの「Step N で…」参照を壊さないため）。マーカーが付いた Step が連続して並ぶことで、§4 を見るだけで進捗が読み取れる状態を保つ。
>
> ```markdown
> ### Step ?: (完了済み) {タイトル}
>
> **状態**: **完了済み** — commit `<id>` で merge（複数コミットに跨る場合は範囲または PR 番号を書く）
>
> ### Step ?: (進行中) {タイトル}
>
> **状態**: **進行中** — {ブランチ名 / セッション ID / 完了予定の sub-step}
> ```
>
> 計画段階で「不採用」と判断した Step は `(不採用)` マーカー + `**状態**: **不採用** — <理由>` で同様に残す（本文は履歴目的で削除しない）。
>
> 実装過程で計画から乖離した場合（API signature 変更 / 採用ライブラリ変更 / 抽象化粒度の変更など）は、該当する設計章（§3 / §5）に **乖離点と理由** を追記してから Step 状態を完了にする。Step 完了マーカーだけ付けて中身の計画と実装がずれた状態を放置しないことで、Step 8 等で DESIGN.md へ転記する際の単一の真実の源として本ドキュメントを使える。
>
> **Step 完了前のサブエージェントレビュー**: 各 Step の実装が `vp check` / `vp test` を通過したら、完了マーカーを付ける前に **サブエージェントで独立レビューする**（`refactoring-plan-template.md` §6 と同じ運用）。レビュー観点は (a) 当該 Step の作業項目すべてに対応する変更があるか、(b) 計画 §3 / §5 と整合しているか・乖離があれば該当節に追記済みか、(c) CLAUDE.md / AGENTS.md の規約に従っているか（コメントは WHY のみ、現在のタスクや修正経緯への言及を書かない、`let` の不使用など）、(d) in-source test が新規ロジック / 退行リスクを網羅しているか、(e) `closest('#doc')` 等のアンカリング不変条件・信頼境界（innerHTML 経由のとき `escapeHtml`）が壊れていないか、を最低限カバーする。指摘事項を反映してから状態マーカーを `(完了済み)` に切り替え、コミットに進む。指摘の重大度に応じて、設計判断にまで踏み込む変更が必要なら §5 の表ごと更新し、軽微な実装修正なら本文と test の更新だけで閉じる。

順序は依存関係順。各ステップ完了で in-source test と手動視覚チェックを通す。

## 5. 設計判断

> プロジェクトの将来の自分 / レビュアー / 他実装者が「なぜこの設計か」を辿るための主要セクション。

## 7. 受け入れ基準

> **書く内容**：このプランが「完了」と判断できる客観条件を 5〜10 項目で列挙する。§1 の対応スコープ表と一貫させる。「視覚回帰なし」「サイズ増分が X 以下」「DESIGN.md が更新済み」のような測定可能な条件にする。

- {規格 / 要件} の N 要件をすべて満たす（§1 冒頭の対応スコープ表が全て ✓）
- 既存挙動の視覚回帰がない（{既存機能} は既存と同等の見た目）
- {サイズ / パフォーマンス} の増分が **{閾値} 以内**
- {既存設計の不変条件 — 例: アンカリング} が壊れない（既存 in-source test 全通過 + 新規追加分も通過）
- {依存機能 — 例: テーマ / ナビゲーション} との連動が DOM 再構築なしで動く
- DESIGN.md §{X} 表の対象行が「準拠 / 実装済み」に書き換わる

## 8. 想定リスクと回避策

> **書く内容**：実装中に発生し得る障害と、それぞれの対策を表にする。レビュー時に「これは想定済み」と確認できる材料にする。
>
> リスクは「外部 API の挙動」「既存設計との干渉」「サイズ / パフォーマンス見積もりのずれ」「信頼境界の漏れ」「Edge case / 環境依存」などのカテゴリで洗い出す。

| リスク                                                | 回避策                                                                                     |
| ----------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| {外部 API が想定どおり動かない}                       | {Step 1 の PoC で検証 / フォールバック設計}                                                |
| {サイズ / パフォーマンスが見積もりからずれる}         | {Step X で実測 / 影響範囲の限定}                                                           |
| {既存機能 — 例: アンカリング / 描画パス との干渉}     | {idempotent 化 / 既存 invariants のテスト追加}                                             |
| {Edge case / 環境依存 — 例: file:// / 特定ブラウザ}   | {フォールバック挙動 / ユーザー通知}                                                        |
| {信頼境界の漏れ — 例: innerHTML 挿入経路}             | {escape / allowlist / CSP との整合確認 / DESIGN.md §11 に追記}                             |
| {自動生成物 / 外部依存のバージョン変更}               | {version pin / 生成ロジックを型エラーで検出可能にする / 生成元バージョンを comment header} |
| {サイレント回帰}                                      | {既知ケースを in-source test に追加 / CI で fail させる}                                   |
| {コミット差分の肥大化 — 自動生成物を commit する場合} | {配布契約上必要であることを明記 / フェールセーフとして ENOENT 案内エラー}                  |
