# Markdown Experience Guidelines

**バージョン 0.1.0 (Draft)**

> **本仕様は策定中である。** セクション番号、要件、準拠レベルは変更され得る。フィードバック、質問、コントリビューションを歓迎する。[GitHub リポジトリ](https://github.com/vercel-labs/mdxg)で Issue を開く、または Discussion を開始すること。

本ドキュメントは **Markdown Experience Guidelines (MDXG)** を規定する。MDXG は、markdown ドキュメントを提示・ナビゲート・編集するときにインターフェースが SHOULD として提供する一連の機能の集合である。

MDXG は文法仕様ではない。CommonMark、GFM、その他いかなる markdown 方言も置き換えない。それらの一段上で動作する —— パース済みの markdown ドキュメントを与えられたとき、ユーザーがそれに対して _何が_ できるべきかを MDXG は定義する。

MDXG は特定のレイアウト、配置、視覚スタイルを規定しない。ツールがこれらの機能をどう表面化するか（サイドバー、パネル、ドロップダウン、ジェスチャー、メニュー、コマンドパレット）は実装上の判断である。本仕様は _何を_ するかを定義し、_どこで_ するかは定義しない。

本ドキュメントにおけるキーワード "MUST"、"MUST NOT"、"SHOULD"、"SHOULD NOT"、"MAY" は RFC 2119 で記述されているとおりに解釈される。

## Motivation（動機）

Markdown は AI エージェントと人間の間の通信におけるデフォルトフォーマットである。主要な LLM はすべて markdown をネイティブに生成する。プラットフォームを跨いで可搬であり、プレーンテキストとしても読め、何千ものアプリケーションでサポートされている。エージェントにとっては理想的である —— トークンコストは低く、生成は簡単で、パースは些細である。

人間にとっては、その体験が追いついていない。ほとんどの markdown インターフェースは、ドキュメントを基本的な描画を施したテキストの平坦なストリームとして扱う。2,000 行の markdown ファイルが 20 行のものと同じように提示される。見出しは文法上は存在するが、構造要素としては無視される。ナビゲーションも、方向感覚も、ドキュメント内の自分の位置の感覚もない。このギャップが、markdown で十分なはずの場面で人々をより重いフォーマットへ追いやっている。

MDXG はこのギャップを埋める。任意の markdown インターフェースが、自分のプラットフォームに合った形でいかようにも実装できる一連の標準機能を定義することで。ゴールは markdown が両方のオーディエンス —— エージェントがそれを生み、人間がそれを消費する —— に対して機能し、その体験が誰にとっても良いものになることである。

## Definitions（定義）

**Document（ドキュメント）**: 単一の markdown ファイル。

**Page（ページ）**: ドキュメントの仮想的な細分。見出し境界によって定義される。

**Page navigation（ページナビゲーション）**: ページ間を移動できるあらゆる仕組み。

**Page outline（ページアウトライン）**: 現在のページ内の見出し構造を示すあらゆる仕組み。

**Preview mode（プレビューモード）**: markdown コンテンツを描画したビュー。

**Source view（ソースビュー）**: 構文ハイライト付きの、生の markdown テキストの読み取り専用ビュー。

**Source edit（ソース編集）**: 構文ハイライト付きの、生の markdown テキストの編集可能なビュー。変更は背後のドキュメントへ同期される。
