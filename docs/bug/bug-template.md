> **このファイルはバグ修正プランのテンプレートです**。新しいバグを起票する際は本ファイルをコピーして `docs/bug-<topic>.md` として配置し、`{プレースホルダ}` を埋める / 引用ブロック内のガイドを削除する形で使う。完了したバグ修正は `docs/archive/bug-<topic>.archive.md` にリネームしてアーカイブする（例: `docs/archive/bug-csp-font-src-missing.archive.md`）。
>
> 設計プラン用テンプレートは `docs/feature/feature-plan-template.md`、リファクタリング用テンプレートは `docs/refactoring/refactoring-plan-template.md` を参照。本テンプレートは「既知の不具合 / 設計と実装の乖離 / spec violation / regression」を起票して fix プランを引くための骨格を提供する。

> **書き方の原則**：
>
> - **再現性を優先**: 「修正前にどうなっているか」を読み手が手元で再現できる粒度で書く。DevTools のどの panel / どの API / どの fixture を使うかまで明記する。silent failure 系の bug は再現確認手順そのものが価値の中心
> - **修正範囲を最小化**: 1 ファイル 1 行で済む変更が複数ファイルに波及している場合は、なぜ波及するかを §1 に書く。波及がないなら §4 を 1 段落で済ませる
> - **受け入れ基準は測定可能に**: 「動くようになる」ではなく「Console violation が 0 件になる」「`document.fonts.load(...)` が fulfilled で resolve する」のような客観条件にする。修正前後で観測値が反転することが分かる粒度
> - **コメントは WHY のみ**: bug 発生経緯 / 過去対応の言及はコードコメントには書かない（コメントポリシーは `AGENTS.md`）。本ドキュメント自体が WHY を残す場

## 3. 再現確認手順

> **書く内容**: 「本当にこの bug が発症しているか」を読み手が手元で確認できる手順を順序立てて書く。DevTools のどのタブ / どの API / どの fixture を使うかまで明記する。検証用 fixture が必要なら §3 末尾に inline で並べる。
>
> 手順は「主確認（最も確実な経路）→ 副確認 → 能動確認（API スニペット）→ 目視確認」の優先順で並べると、CI 自動化への移行がスムーズ。Network パネル / Performance / Sources 等、使えない経路があるなら「使えない理由」を末尾の注で明記する（読み手が「そこを見てしまう」のを構造的に避ける）。

1. {初期セットアップ：ビルド / 環境準備}
2. {再現対象を開く / 操作する手順}
3. **主確認 ({DevTools の特定 panel / API})**: {期待される観測}（例: Console に違反メッセージが N 件出る、特定 element が見えない、computed value が想定外、等）
4. **副確認 ({別の経路})**: {期待される観測}
5. **能動確認 (Console から実行)**: {実行コード スニペット} を実行し、{期待される return / reject / 観測値}：

   ```js
   // 検証コード例
   ```

6. **目視確認**: {視覚的な変化 / レイアウト / フォント / 色などの観測点}

注: {DevTools / API の罠 / 使えない経路があれば補足。例: 「Network パネルは inline 化済みリソースを追跡しない」}。

## 5. 受け入れ基準

> **書く内容**: §3 再現確認手順と対称な形で、修正後にどの観測値が反転するかを列挙する。客観的に検証可能な条件にする。「修正前 N 件 → 修正後 0 件」「修正前 reject → 修正後 fulfilled」のような数値 / Promise status / boolean で書ければ理想。

- {修正対象ファイル / 設定}に修正点が反映されていること
- ビルド成果物（{`dist/<path>` 等}）に修正が伝播していること
- §3 検証 fixture / 操作で {主観測値} が {期待値} になる（例: Console に CSP violation が **0 件**）
- §3 能動確認 ({API}) で {期待値}（例: `document.fonts.load(...)` の Promise がすべて `fulfilled`）
- {目視確認の項目があれば}（例: 数式が KaTeX 公式フォントで描画される）
- 既存 `vp test` / `vp check` がすべて通過する（修正による副作用がない）

{受け入れ基準から外す項目があれば 1 段落で。例: 「未使用 family の status が `'unloaded'` のままなのは仕様通りで bug 修正の指標にならない」}。
