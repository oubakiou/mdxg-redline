## 14.2 翻訳辞書とランタイム

辞書を **UI / CLI で物理分割** することで bundle 重複を避け、ランタイム実装も `document` / `localStorage` 依存の有無で **3 レイヤーに分割** する：

- **辞書 (UI 用)**: `src/app/i18n/messages.{en,ja}.ts`（UI 辞書 約 170 entry、`online.*` / `footnote.*` 含む）。en 側が `MessageKey` を export し、ja 側は `satisfies Record<MessageKey, string>` で key 一致を tsc レベルで保証する。ブラウザ bundle のみが import
- **辞書 (CLI 用)**: `src/cli/i18n/messages-cli.{en,ja}.ts`（`cli.*` 約 36 entry、help は block 形式）。CLI bundle のみが import
- **純粋ロジック (Node / ブラウザ共通)**: `src/core/i18n/i18n-core.ts`。`translate` / `translatePlural` / env・navigator からの言語判定 / 初期 lang 解決を担い、`document` / `localStorage` に触らない
- **ブラウザ副作用**: `src/app/i18n/i18n-browser.ts`。`setLang` / `getLang` / `applyI18nDataset` / `subscribeLangChange` 等の wrapper で `<html lang>` / `localStorage` / `document` を操作する
- **CLI 副作用**: `src/cli/i18n.ts`。`setCliLang` / `getCliLang` / `translateCli`。`document` / `localStorage` には触らない

UI / CLI それぞれ独立した import tree なので bundle 重複なし。online 版 (`dist/hosting/index.html`) は standalone HTML から派生して生成され、UI 辞書は派生元に既に inline 済みなので **online 版での辞書再 inject は不要**（[build-pipeline.md §13](./build-pipeline.md#i18n-経路の責務分離)）。

## 14.3 言語決定の優先順位

CLI 自体と review HTML は **責務分離** し、互いに影響しない。CLI は HTML 側に lang 関連属性 (旧設計案の `<html data-lang-init>` 等) を一切埋め込まず、HTML 側は `localStorage > navigator.language > 'en'` で独立に決定する。この責務分離が本実装の i18n の正典で、§14.1 / §14.4 / build-pipeline.md §13 の記述はすべてこの規則に従う：

| 経路                                                   | 優先順位                                                              |
| ------------------------------------------------------ | --------------------------------------------------------------------- |
| CLI 自体 (help / stderr)                               | `--lang` フラグ > `$LC_ALL` > `$LC_MESSAGES` > `$LANG` > `'en'`       |
| review HTML 起動時 (CLI 生成 standalone / online 共通) | `localStorage('mdxg-redline.lang')` > `navigator.language` > `'en'`   |
| 起動後の toggle                                        | toggle で `setLang(lang)`、`localStorage` に保存 (次回起動時の最優先) |

env → lang のマッピング (CLI 自体にのみ適用、POSIX 階層 `$LC_ALL > $LC_MESSAGES > $LANG`)：

| 入力                                                   | 扱い                                                             |
| ------------------------------------------------------ | ---------------------------------------------------------------- |
| `undefined` / 空文字 (`""`)                            | **未設定として skip**。例: `LC_ALL=""` なら `LC_MESSAGES` を見る |
| `ja`, `ja_JP`, `ja_JP.UTF-8`, `ja-JP` (`^ja(_\|-\|$)`) | `ja` で確定 (後段は見ない)                                       |
| `en_US.UTF-8` / `C` / `POSIX` / その他                 | `en` で確定                                                      |

階層全てが未設定 / 空文字の場合は最終 fallback として `'en'`。`navigator.language` も同じ正規表現で en / ja のみ区別、それ以外は `'en'`。

## 14.4 CLI オプション

`--lang <auto|en|ja>` を **サブパーサ非依存のグローバル メタフラグ** として扱い、bootstrap で argv strip + lang 抽出 + エラー検出を 1 パスで行う。サブパーサ (run / clean) は `--lang` の存在を知らずに動く。

| 値     | 挙動                                                              |
| ------ | ----------------------------------------------------------------- |
| `auto` | env から推定（マッピングは §14.3）。env も空なら `en` で CLI 出力 |
| `en`   | CLI の help / stderr を英語で出力                                 |
| `ja`   | CLI の help / stderr を日本語で出力                               |

不正値 (`fr` / `spec.md` 等) は **二段階解析** で扱う：(a) lang 抽出が `{lang, argv, error}` を返し、(b) main 側で「help 最優先 → lang エラー reject → 通常モード判定」の順に処理する。`--lang fr --help` のような lang エラーと help の同時発生では help が勝つ (既存の help 最優先契約を維持)。後勝ち + 不正値の保持: `--lang fr --lang en` → `lang=en, error=invalid 'fr'`（「不正値検出後に解析を停止」モデルと整合）。

**help 最優先の構造的保証**: lang 抽出は次トークンが `--` で始まる場合だけでなく、短形式の `-h` を含む `HELP_FLAGS` セットにヒットした場合も「値欠落」として扱い、その次トークンを **argv から消費せずに保持** する。これにより `--lang --help` / `--lang -h` / `--lang --clean` 等の組み合わせでも help / clean モード判定が `lang error reject` より構造的に先に通る。`-h` のような `--` prefix を持たない短形式が `invalid_value` として誤消費される退行は test で固定する。

`--lang` は **CLI 自体の help / stderr 出力にのみ作用** し、生成 HTML の初期言語には影響を与えない（HTML 側の決定は §14.3）。

## 14.5 翻訳キー命名規約と複数形分岐

13 名前空間 (`toolbar.*` / `comments.*` / `modal.*` / `toast.*` / `empty.*` / `page_nav.*` / `search.*` / `dialog.*` / `diagram.*` / `online.*` / `footnote.*` / `mobile.*` / `cli.*`) に整理 (総数約 205)。

suffix 命名規約：

| suffix                      | 用途                                                                 |
| --------------------------- | -------------------------------------------------------------------- |
| (なし)                      | 要素の `textContent` 用 key (`data-i18n` で挿す)                     |
| `_aria`                     | HTML 属性 `aria-label` 専用 (`data-i18n-aria-label` で挿す)          |
| `_tooltip`                  | HTML 属性 `data-tooltip` 専用 (`data-i18n-data-tooltip`)             |
| `_placeholder`              | HTML 属性 `placeholder` 専用 (`data-i18n-placeholder`)               |
| `_zero` / `_one` / `_other` | 複数形分岐 (`translatePlural({baseKey, count, params?})` で内部解決) |

複数形分岐の base key は **4 件 + 単一 key 1 件** (`comments.count_label` / `toast.render_failed` / `modal.confirm_delete_comments` / `search.count` + `toast.feedback_written`)。`Intl.PluralRules` は使わず `if (count === 0) ... else if (count === 1) ... else ...` の素朴ロジックで実装し、`_zero` → 辞書未登録なら `_other` に fall back する。

`data-i18n` は **leaf 要素のみ** に付与する規約 (textContent 代入で子要素が破棄されるため)。複合構造 (テキスト + caret icon 等) では、テキスト部分を専用 leaf `<span data-i18n="...">` で wrap する。aria-label / placeholder / title / data-tooltip は属性なので複合構造の親要素に直接付与可能。dev mode では `applyI18nDataset` が `[data-i18n]` 要素の `children.length > 0` を検出して `console.warn` (regression 検出)。具体例：

```html
<!-- NG: button が caret span を子要素として持ち、applyI18nDataset の textContent 代入で caret が消える -->
<button data-i18n="toolbar.open">Open <span class="menu-caret" aria-hidden="true">▾</span></button>

<!-- OK: テキスト部分のみ leaf span で wrap し、caret は兄弟として残す -->
<button>
  <span data-i18n="toolbar.open">Open</span>
  <span class="menu-caret" aria-hidden="true">▾</span>
</button>
```

## 14.6 DOM 連携と動的 UI の追従

HTML markup には `data-i18n="comments.empty"` 等の dataset 属性を持たせ、起動時に `applyI18nDataset(document)` が DOM を 1 回 walk して各属性を翻訳する。walk セレクタは `[data-i18n], [data-i18n-aria-label], [data-i18n-placeholder], [data-i18n-title], [data-i18n-data-tooltip]` の和、`#doc` 配下は `closest('#doc')` で除外する (アンカリングの textContent 計算 DESIGN.md §6 を保護)。

**`#doc` 配下の構造的例外**: `[data-footnote-backref][data-i18n-aria-label]` (脚注 backref の `↩` リンク) は `text-segment-skip-rules.ts` で既にアンカリング skip 対象に登録されているため、`#doc` 配下でも walker の包含対象とする。ただし `applyToElement` の `isInsideDocProtectedRoot` ガードは生きていて、textContent (`↩`) は不変のまま **aria-label のみが翻訳** される (二重防御で本文 textContent シーケンスを保護)。将来同様の例外を増やす場合も「`text-segment-skip-rules.ts` でアンカリング skip + 属性のみ翻訳」の二条件を満たすことが必須。

**setLang の処理順** (`i18n-browser.ts` 専用、CLI 側は呼ばない)：

1. module-local state を更新
2. `<html lang>` を更新
3. `applyI18nDataset(document)` で静的 markup を再描画 (DOM 再構築なし)
4. `subscribeLangChange` のリスナを呼んで動的生成済み要素を再描画
5. `writeStoredLang(lang)` で `localStorage` に保存 (**最後**。storage 障害時もセッション内 toggle が機能する)

**動的 UI の購読パターン**: 文書を複数回読み込む / モーダルを複数回開閉する等で同じ生成元関数が再呼び出しされるケースで `subscribeLangChange` を毎回呼ぶと listener が累積する。これを防ぐため、各動的 UI モジュールは **state の有無で 2 つのパターンに分かれる**：

- **4 関数パターン (state あり)**: `setup` / `show` (state 更新) / `clear` (state null) / `teardown` (全 subscription 解除 + state null + DOM 初期化) を持つ。表示中の文言が言語切替に追従するために state を保持する必要がある動的 UI に適用 (現在表示中の URL / エラー key / renderer など)
- **2 関数パターン (state なし)**: `setup` / `teardown` のみ。`subscribeLangChange` 内で module-local singleton (app-state 直結) を都度読んで再描画する動的 UI に適用。state を二重に持たないため `show` / `clear` 相当は不要

どちらのパターンでも `setup` は **idempotent guard** (`if (langSubscription !== null) return`) で二重購読を防ぐ。**運用前提**: 実 UI では bootstrap で 1 回 `setup` した後 CLI 終了まで生存するため `teardown` は本番経路で呼ばれない。teardown を残す理由は (1) test fixture が module-local state を巻き戻して別 fixture と独立させる、(2) HMR / 将来の SPA 風文書切替で古い loader / listener が残らないようにする、の 2 点。これらの利用形態が無くなった場合のみ teardown を削除してよい。

**文書ライフサイクル hook**: 各入力経路 (Open file / Paste markdown / online fetch / embedded MD) で `clearOnlineSource()` / `showOnlineSource(url)` を個別に呼ぶと、新しい入力経路を追加した時に呼び忘れる構造的リスクがある。`src/app/document/load-document.ts` の `createDocumentLoader(baseLoader)` factory で `DocumentSource = { kind: 'local' | 'online'; ... }` を引数に取る `loadDocument` と購読 API `registerOnDocumentLoad` を組み立て、全 callsite を `loadDocument({kind, ...})` 経由に統一する。hook の失敗隔離 / 反復中 unsubscribe 耐性 / Unsubscribe 戻り値による個別解除は同モジュールの実装で担保する。`app-wiring.ts` の `resolveBootstrapDeps` で online decorator (Mermaid / KaTeX / Shiki lazy fetch) 適用済みの `loadFromMarkdown` を `createDocumentLoader` に渡す単一合成ポイントを置くことで、decorator バイパスを構造的に防ぐ。

## 14.7 サイズ見積もり

| ビルド                       | 増分 (gzip)                     |
| ---------------------------- | ------------------------------- |
| `dist/embed-template.html`   | +約 3 KB (辞書 2 言語 + helper) |
| `dist/standalone.html`       | +約 3 KB                        |
| online build (`mkdn.review`) | +約 3 KB                        |

辞書 2 言語の raw size は約 6–7 KB、helper (`i18n-core` / `i18n-browser`) は 1 KB 未満。対応言語数が 3 以上に増えた段階で「全言語を HTML に常時 inline」方式は配布物サイズを線形に押し上げるため、Shiki grammar / Mermaid runtime 等と同じく「必要な言語だけ後から fetch する」方式 (lazy load / 別ファイル分割) への移行を検討する。本実装は 2 言語のみのため inline + 設定モーダル内 select box を採用。
